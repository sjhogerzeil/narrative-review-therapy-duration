# Reproducing or adapting this review

This repository contains the complete evidence base, methodology, and audit trail for a structured narrative review on the duration of healing from complex developmental trauma. It is designed to be forked, reproduced, and adapted.

## What you can do with this project

**Reproduce the review.** Read the same sources, follow the same methodology, and check whether you arrive at the same conclusions. The structured annotation template, extraction tables, and synthesis method are fully documented. Every claim in the discussion can be traced back through the reference chain to a specific source annotation.

**Challenge the review.** The counterpoint layer (Layer 3) was designed to contain the strongest evidence against the working thesis. If you believe the counterpoint was not engaged honestly, or if you know of evidence that was missed, the methodology provides a clear path: annotate the source using the template, extract data into the relevant dimensions, and see whether the synthesis survives.

**Adapt the methodology for a different question.** The CIS framework, six-layer evidence architecture, structured annotation template, and checkpoint-gated AI workflow are not specific to therapy duration. They can be applied to any question that spans heterogeneous evidence types and requires interpretive synthesis rather than statistical aggregation.

## How the project is structured

```
1_introduction/     Research question, population, healing definition, stance
2_methods/          Design, search strategy, annotation protocol, synthesis method, AI methodology
3_results/
  sources/          89 annotated source notes (one .md per source, organised by layer)
  primary/          Extracted data by primary synthesis dimension (11 files)
  secondary/        Extracted data by secondary question (6 files)
4_discussion/       Synthesis + 8 discussion sections (~30,000 words)
5_tables_and_figures/  5 tables + 2 figures (.md + .csv)
6_references/       APA 7 bibliography (auto-generated by sync.sh)
_index/          Semantic index (auto-generated by sync.sh)
```

Read `CLAUDE.md` for the full orientation, including the task routing table that tells you which files to read for any given task.

## Reproducing the review

### Step 1: Understand the methodology

Read these files in order:

1. `2_methods/1_design.md` — CIS methodology and six-layer rationale
2. `2_methods/2_search_strategy.md` — how sources were identified
3. `2_methods/9_annotation_protocol.md` — how sources were annotated
4. `2_methods/11_synthesis_method.md` — how findings were synthesised
5. `2_methods/13_ai_methodology.md` — how AI was used and controlled

### Step 2: Verify the source annotations

Each source annotation in `3_results/sources/` follows a structured template with YAML frontmatter. The annotation contains extracted data, cross-cutting observations, and verbatim quotes with citations. You can verify any annotation against the original source.

Note: some sources are behind paywalls or are published books. The `todo.md` file lists inaccessible sources and their impact on the conclusions. The annotations for these sources were created from abstracts, secondary descriptions, or partial access — this is noted in each annotation's metadata.

### Step 3: Verify the extraction

The extraction tables in `3_results/primary/` and `3_results/secondary/` aggregate data from source annotations by synthesis dimension. Each entry includes a source ID that links back to the annotation. You can check whether the extraction accurately represents what the source says.

### Step 4: Verify the synthesis

The synthesis (`4_discussion/1_synthesis.md`) is the review's central argument. Every claim includes back-references to sources. The argument's structure — the 4-factor framework, the endpoint-duration nexus, the six trajectory shapes — can be checked against the extraction tables and source annotations.

### Step 5: Run the consistency checker

```bash
./_scripts/sync.sh check    # dry run — reports missing citations, unindexed sources
./_scripts/sync.sh all      # regenerates references, index, and layer READMEs
```

### Step 6: Use git tags to navigate the audit trail

The repository is tagged at key milestones. You can check out any tag to see the project state at that phase:

| Tag | Phase |
|-----|-------|
| `iteration-0-initial` | Initial research question and definitions |
| `v0.1-setup` | Project structure and methodology |
| `v0.2-search-complete` | All sources identified and annotated |
| `v0.3-reddit-added` | Reddit client narratives added (Layer 5) |
| `v0.4-extraction-complete` | Data extraction into primary/secondary dimensions |
| `v0.5-methodology-revised` | Methodology updated after Phase 1 insights |
| `v0.6-synthesis-revised` | Synthesis rewritten with 89 sources |
| `v0.7-discussion-complete` | All 9 discussion sections drafted |
| `v0.8-tables-figures` | Tables and figures added |

```bash
git tag                          # list all tags
git log --oneline v0.2..v0.6    # see what changed between search and synthesis
git diff v0.4 v0.6 -- 4_discussion/1_synthesis.md   # how synthesis evolved
```

## Forking for a new question

If you want to use this methodology for a different research question:

### Step 1: Fork the repository

Keep the directory structure and tooling. Delete the content in `3_results/`, `4_discussion/`, `5_tables_and_figures/`, and `6_references/`.

### Step 2: Rewrite the introduction

Define your research question, population, working definition of the key concept, and your stance (honest advocacy, neutral, or critical). These go in `1_introduction/`.

### Step 3: Design your evidence layers

The six-layer architecture is not fixed. Your question may need different layers. The principle is: each layer represents a different epistemological perspective on the same question, so that findings can be triangulated across evidence types. Design layers that capture the heterogeneity of evidence relevant to your question.

### Step 4: Adapt the search strategy

Each layer needs its own search plan with source budgets, search terms, and stopping rules. See `2_methods/2_search_strategy.md` for the template.

### Step 5: Keep the annotation template

The structured annotation template (`3_results/sources/SOURCE_TEMPLATE.md`) works for any source type. Adapt the YAML fields and extraction dimensions to your question, but keep the structure: metadata, annotation body, cross-cutting observations, verbatim quotes.

### Step 6: Design your synthesis dimensions

The primary and secondary dimensions should be derived from your research question. See `2_methods/11_synthesis_method.md` for the analytical process. Expect emergent dimensions — CIS methodology anticipates that the predefined categories will evolve during the review.

### Step 7: Publication and discoverability

Before publishing the repository, generate the files that make the review findable by humans, search engines, and AI systems:

1. **`README.md`** — executive summary with structured metadata block (population, method, finding, framework) at the top. This is the primary entry point for all readers and crawlers.
2. **`CITATION.cff`** — machine-readable citation metadata. GitHub renders a "Cite this repository" button from it. Include ORCID. Update the DOI after Zenodo archival.
3. **`llms.txt`** — AI-specific orientation file ([llmstxt.org](https://llmstxt.org/)). Contains: citation (APA + BibTeX), key metadata, key findings as numbered list, file map. Optimized for LLM context windows, not human reading. Update after each major revision.
4. **`codemeta.json`** — rich machine-readable metadata indexed by Google Dataset Search, OpenAIRE, and DataCite. Generate from CITATION.cff or write manually.
5. **GitHub repo settings** — set the description (one-liner with the key finding) and topics (keywords for indexing). These are set via GitHub UI or `gh repo edit`.
6. **`_scripts/build_pdf.sh`** — generates a printable PDF from the manuscript chapters. Attach the PDF to GitHub Releases.

These files should be regenerated when the review is substantially revised. Add `llms.txt` and `codemeta.json` to `sync.sh` if you want automated updates, or maintain them manually at publication milestones.

### Step 8: If using AI assistance

Read `2_methods/13_ai_methodology.md` for how AI was used in this project. Key principles:

- **Checkpoint-gated workflow.** Do not let the AI proceed from one phase to the next without human review. The critical checkpoints are: after source annotation (did the AI read the sources correctly?), after synthesis (is the argument warranted by the evidence?), and after discussion (are the qualifications honest?).
- **Structural counterpoint.** Design at least one evidence layer specifically to challenge your working thesis. This is a safeguard against confirmation bias that is at least as important when using AI as when working alone.
- **Audit trail.** Use version control. Commit after each phase. Tag milestones. The git history is your audit trail.
- **Disclose fully.** Follow ICMJE 2026 Section V and COPE guidelines. Specify the model, the task distribution, and the quality control mechanisms. See `2_methods/13_ai_methodology.md` for the format.

## Ethics

### Forum and social media data

This review includes data from public online forums (Reddit, MyPTSD.com). These posts are cited as sources — referenced by username and thread title — not scraped as research data. The posts are publicly accessible and were accessed through standard web browsing or API search. No private messages, locked threads, or deleted content was used. Usernames are included because they are part of the public record and are necessary for source traceability.

Researchers adapting this methodology for questions involving more sensitive populations or private forums should consider whether additional ethical review is appropriate.

### Full-text PDFs and copyright

This repository includes PDF full texts of journal articles in `3_results/sources/[layer]/_fulltext/` directories. These are included to support replicability — allowing reviewers to verify annotations against the original sources. Their inclusion is intended as fair use for research and educational purposes.

If you are a rights holder and object to the inclusion of a specific file, please open an issue and the file will be removed promptly. The corresponding source annotation (which contains only extracted data, not the full text) will remain.

Not all sources have PDFs archived in the repository. Sources accessed via the web (practitioner blogs, forum posts, podcasts) are referenced by URL. Books are referenced by citation. The `access` field in each source note's YAML frontmatter indicates how the source was accessed: `full`, `abstract-only`, `provided-manually`, or `inaccessible`.

### Inaccessible sources

Several sources identified as relevant could not be accessed due to paywalls or publication format (books). These are documented in `7_process/todo.md` with an assessment of their impact on the conclusions. The review does not claim to be exhaustive — CIS methodology uses iterative, question-driven search rather than comprehensive screening — but inaccessible sources represent genuine gaps in the evidence base that are noted honestly.
